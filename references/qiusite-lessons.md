# 丘斯特项目经验

这些经验来自丘斯特/石头 Web 项目，可泛化到前端生成类项目。

## 架构经验

- 生成项目不能只服务第一张截图，要天然支持下一批页面。
- Route page 负责组合；共享组件负责重复 UI；纯 helper 负责展示/业务规则。
- 数据要容易编辑，但不要散落在 JSX。
- 纯前端项目也要有清晰静态/模拟数据契约，方便未来接后端。

## 页面系统经验

丘斯特是移动 App-like 电商：Tab 页面、详情页、账号页、流程页并存。可泛化为：

- 一个共享 app/site 外壳。
- route group 承载共同导航。
- detail route 承载详情页。
- form/workflow route 承载结算、联系、预约、设置、 onboarding。
- 帮助、政策、说明页作为支撑页面。

## UI 经验

- 容器要稳定：图片比例、导航高度、抽屉高度、底部 padding 都要明确。
- 少量可复用 UI primitives 胜过每页单独写样式。
- 空状态和兜底状态属于设计的一部分。
- H5 项目注意 `100dvh` 和 safe-area。
- 文案要短、明确、行动导向。

## 持续维护经验

丘斯特靠这些边界保持可维护：

- `app/` 下放路由页面。
- `components/` 放共享 UI。
- `lib/` 放纯规则。
- 业务域复杂交互放独立目录。
- docs/specs 记录关键决策。

新前端项目可对应为：

- `components/`：共享组件。
- `data/`：可编辑内容。
- `lib/`：公共 JS/纯函数。
- `docs/frontend-map.md`：给后续开发者和 agent 看。

## 避免什么

- 一个巨大的 `page.tsx`。
- 多个页面重复写同一组数组。
- 所有组件都 `'use client'`。
- 每页发明一套无法复用的 class。
- 前端需求还没证明必要，就提前做后端式复杂抽象。
